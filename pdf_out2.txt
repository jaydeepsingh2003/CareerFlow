AI-Powered Adaptive Onboarding Platform
Static, one-size-fits-all onboarding wastes time. The goal is to build an AI engine that reads a new hire’s
resume and job description, identifies the skill gaps, and generates a customized training path. This idea is
already showing up in recent AI-education projects – for example, one open-source system “generates
personalized learning paths powered by AI” with a React+Flask stack , and tools like ResuMasters extract
resume skills and suggest upskilling . In our solution, we’ll harness free data (like the O*NET skills
database ), open-source models and APIs, and a clean web interface to deliver a career-specific learning
roadmap. Below is a full breakdown of requirements, related work, proposed features and architecture, an
implementation timeline with tech choices, and a prototype plan using free resources.
Requirements Summary
The hackathon specifies an AI-driven learning engine with these core capabilities (summarized from the
provided brief):
Skill Extraction (Intelligent Parsing):  Parse a resume and a job description to extract key skills and
proficiency levels. (E.g. NLP or LLM-based parsing to list skills from each document.)
Gap Analysis & Mapping:  Compare the candidate’s skills to the job’s required skills, identifying the
“skill gap.” Then dynamically generate a personalized learning pathway  that fills those gaps. 
Web Interface:  A minimal but clear web UI where a user uploads a resume and JD and sees the
resulting custom training roadmap visualized. (This must be functional – e.g. a simple React/HTML
form with results.)
Adaptive Logic:  Implement original adaptive logic (graph-based or knowledge-tracing algorithm) to
sequence the training modules for optimal learning. Pretrained models can be used for language
tasks, but the path-planning logic must be our design.
Traceability:  Provide an optional reasoning trace or log of how the system made its decisions
(important for transparency per the criteria).
Evaluation Goals:  The solution should reduce redundant training and cover role-specific content,
work across different job categories, avoid AI “hallucination” by grounding in real curricula, and be
well-documented for judges.

gap-analysis step. ResuMasters shows how to leverage GPT (or similar) to match resumes to job
requirements and suggest improvements.

Proposed Features & Architecture
We propose a modular system with the following unique features and components:
Document Parsing and NLP:
Use OCR/Text extraction (PyMuPDF or PDF libraries) for resumes in PDF/Word. 
Apply NLP (spaCy, Hugging Face Transformers, or fine-tuned models) to identify relevant phrases:
skills, certifications, education, job titles, and experience levels. Pretrained NER or keyword matching
(with tech skill lexicons from O*NET) can tag skills in text. 
Similarly parse the target job description to list required skills, responsibilities, and role level.
Skill Matching & Gap Identification:
Compare the two skill sets. Compute which required skills the candidate lacks or rates lower . This can
be a fuzzy match (embedding similarity) rather than exact text. (For example, use a
SentenceTransformer to embed skills and calculate similarity between candidate skills and JD skills.) 
The output is a skill gap list.  Each missing skill or underdeveloped area is a node we need to address.
Adaptive Path Planning:
Knowledge Graph Model:  Build or use an existing skill graph (e.g. from O*NET or curated) where
nodes are skills/modules and edges indicate prerequisite relationships. Each job role might map to
certain goal nodes. We can then compute an optimal path through this graph that covers all gaps
with minimal redundancy. Graph algorithms (like shortest path, topological sort) can sequence the
modules logically. 
Graph-Based or Tracing Algorithm:  As suggested, we might implement a graph search or
knowledge-tracing algorithm. For instance, treat each missing skill as a goal, then plan a sequence of1

LLM-assisted Sequencing:  Optionally, we can prompt an LLM (GPT-4o/GPT-3.5 or an open model)
with the list of gaps and ask it to propose an ordered curriculum drawn strictly from our course
catalog. But the core logic should rely on our graph model to avoid hallucinations.
Course/Curriculum Mapping:
Maintain or retrieve a catalog of training resources (courses, videos, tutorials) tagged by skill. This
could be a simple database of (skill → recommended course) entries. We could seed it with open
resources (e.g. top Coursera/edX/Udemy courses, or free YouTube playlists) and align them to
O*NET skills. 
To avoid cost, we’ll focus on free courses or content: YouTube tutorials, free MOOCs (audit mode),
open textbooks, etc. 
The system maps each gap skill to a module or a set of resources. This could be done via keyword
matching or embeddings between the skill and course descriptions.
Functional Web Interface:
Frontend:  A simple React or Next.js app (hosted free on Vercel/Netlify) with an upload form for
resume and JD. Display the resulting learning path in a clean list or graphical timeline. Possibly allow
download of the plan PDF. 
Backend/API:  A lightweight REST API (Python Flask/FastAPI or Node.js Express) on a free tier host
(e.g. Render , Railway, or Heroku free tier) that handles the parsing and computation. 
The frontend calls the API (e.g. POST files), then polls for results (or waits synchronously if quick). The
backend returns JSON of skill gaps and recommended modules. 
We’ll  visualize  the  learning  path  as  a  sequence  of  steps  (e.g.  numbered  modules  with  titles,
descriptions, and links). A minimal dash or line chart could show progress. Emphasize clarity and
ease of use to score well on UX.
Traceability/Logging:
Keep an internal log of decisions (e.g. “Skill X mapped to Course Y because…”). We can expose these
as a toggle or in a “details” panel. This satisfies the “reasoning trace” criterion. 
Ensure all external data/model sources (datasets, LLMs) are documented per the rules.
Cross-Domain Scalability:
The skill matching should not be hardcoded for one field. By using O*NET (a universal job-skill
taxonomy ) and embedding models, our system can handle technical, desk, or manual roles. We’ll
validate with a few varied examples (e.g., software engineer vs. warehouse operator).

Implementation Roadmap (with Tech Stack & Timeline)
Data Gathering (Week 1):  Download the O*NET database  (free skills taxonomy), and fetch a
resume corpus (Kaggle has free resume datasets). Gather sample job descriptions (Kaggle jobs
dataset or scraped). 
Output:  Local database of skills, some example resumes/JDs for development.
Skill Extraction Prototype (Weeks 2–3):
Build a basic parser: use Python (spaCy or NLTK) or Hugging Face NLP to identify skills. If time, fine-
tune a named-entity model on resume/skill labels or prompt an LLM to list skills. 
Test on the sample data. Iterate until the parser reliably pulls out known skill terms. 
Output:  A function/API endpoint extract_skills(text) → [skill, level] .
Gap Analysis Engine (Week 4):
Implement logic to compare two skill lists (resume vs JD). Use embeddings (SentenceTransformers)
to catch synonyms (e.g., “JS” vs “JavaScript”). 
Generate a ranked list of missing skills or weak areas. 
Output:  A JSON of identified skill gaps for each input pair .
Curriculum Mapping (Weeks 5–6):
Build or source a small catalog of training modules (e.g., key online courses or modules tagged by
skill). This could be a CSV or database table. 
Develop the adaptive pathing: create a simple graph of prerequisites (could start with linear: one
course per skill). Then improve by linking related skills (perhaps by O*NET’s “Skill-Level” data). 
Optionally call an LLM in a limited way: e.g. “Given skill gaps X, Y, Z, list the top 5 free courses to learn
these.” Use its output but cross-check with known courses. 
Output:  A mapping function plan_learning_path(gaps) → [ordered modules] .

Prototype Plan & Free Resources
To keep it free, we’ll use open-source tools and data:
Models & APIs:  Use free-tier LLMs (GPT-3.5-turbo via OpenAI’s free credits or Llama 3 / Mistral via
Hugging Face). SentenceTransformers (all free) for embeddings. No paid licenses.
Data:  O*NET database (free CC license)  for skill taxonomy. Kaggle’s free resume and jobs
datasets. Course/resource info scraped from public sites (e.g., Wikipedia lists of courses, YouTube
channels).
Hosting:  Vercel/Netlify for frontend (free static hosting). Render or Railway free tier for backend.
GitHub Pages could serve static parts.
Dev Tools:  Python (open-source libraries), Node/JavaScript (React), GitHub for version control,
GitHub Actions for CI (all free).
Visualization:  Libraries like Chart.js or D3 (open-source) if needed for path UI. 
Workflow:
Build a GitHub repo, track issues for tasks. 
Code the backend logic and deploy as an API on Render (their free plan supports small apps). 
Develop the React UI locally, then deploy on Vercel (free hobby plan) or Netlify.
Document everything in README (setup, usage, data sources, algorithm overview).
Prepare a 2–3min demo video (screen-record the UI, highlight how paths adapt to different input)
and a 5-slide deck (outline solution, architecture, stack, algorithms, datasets).
